# Entry Signal 設計プロセス標準

## 1. 本文書の目的

新しい Entry Signal を設計・実装する際に、
リサーチ → 検討 → 設計 → 計画書作成 の各段階で
何を行い、何を成果物として残すかを体系化する。

Orderly Pullback Entry の設計過程で得られた知見を一般化したもの。

---

## 2. プロセス全体像

```
Step 1: セットアップ定義
    ↓
Step 2: プール構築設計
    ↓
Step 3: 評価軸設計（maturity / timing / risk_reward）
    ↓
Step 4: 統合とスコアリング設計
    ↓
Step 5: システム整合性検証
    ↓
Step 6: 実装計画書作成
    ↓
Step 7: 実装・検証
```

各ステップの成果物が次ステップの入力になる。
途中で前ステップに戻ることは正常であり、
特に Step 3 と Step 5 の間で反復が発生しやすい。

---

## 3. Step 1: セットアップ定義

### 目的

この Entry Signal が対象とするセットアップとは何かを、
トレーディング戦略の観点から明確に定義する。

### 実施内容

#### 1.1 セットアップの構造記述

以下を自然言語で明確に記述する。

- どのようなトレンド状態の銘柄を対象とするか
- どのような価格パターンがセットアップを構成するか
- セットアップが成立する条件と、崩壊する条件は何か
- 理想的なエントリポイントはどこか
- なぜそのエントリポイントが良いのか（期待値の根拠）

#### 1.2 既存 preset との対応関係

対象セットアップに近い既存の preset を特定する。
完全一致は不要だが、以下を明確にする。

- どの preset の候補がこのセットアップの母集団になりうるか
- 既存 preset の条件のうち、セットアップ定義に含まれるものと含まれないもの
- 既存 preset がカバーしない、このセットアップ固有の側面

#### 1.3 時間的特性の分析

セットアップ検出からエントリ最適日までのタイムラグを推定する。

- 検出とエントリが同日になるタイプか
- 数日のラグがあるタイプか
- ラグがある場合、典型的な日数範囲は何日か

この分析が detection_window_days の根拠になる。

### 成果物

「セットアップ定義書」：上記 1.1–1.3 の記述。

---

## 4. Step 2: プール構築設計

### 目的

Entry Signal 専用の判定対象プールをどう構築するかを決定する。

### 実施内容

#### 2.1 preset_sources の選定

Step 1.2 で特定した preset から、
pool の起点として使用する preset を選定する。

判断基準:
- そのpresetのduplicate判定が、セットアップの検出として十分か
- 複数 preset を含めるべきか（類似セットアップの統合）
- 含めるべきでない preset はないか（異なる文脈の混入リスク）

#### 2.2 detection_window_days の決定

Step 1.3 の時間的特性分析に基づいて決定する。

原則:
- セットアップの 90 パーセンタイル持続期間をカバーする
- 長すぎる window はセットアップの鮮度を劣化させる
- 短すぎる window はエントリ機会を逃す

#### 2.3 セットアップ無効化条件の定義

セットアップの前提が崩壊する条件を列挙する。

各条件について:
- 使用するフィールド名
- 条件の種類（below / above / at_or_below 等）
- 閾値または参照先
- なぜこの条件でセットアップが壊れるかの根拠

原則:
- 少なすぎると壊れたセットアップが pool に残る
- 多すぎると健全なセットアップが早期に無効化される
- 最初は保守的に（少なく）設定し、検証データから追加する

#### 2.4 snapshot_at_detection フィールドの決定

pool 登録時にスナップショットとして保存するフィールドを列挙する。

必要な基準:
- 無効化判定の文脈情報として後から参照するもの
- 成熟度評価の基準線として使うもの（検出時との差分計算）
- R/R 評価の参照点として使うもの

不要な基準:
- 毎日の enriched snapshot から直接取得できるもの（snapshot 不要）
- ただし、日が経つと値が変わるもの（rolling 系）は固定が必要

#### 2.5 pool_tracking フィールドの決定

pool active 期間中に日次更新するフィールドを列挙する。

典型例:
- `low_since_detection`: R/R の SL 参照
- `high_since_detection`: R/R の TP 参照

### 成果物

「プール構築定義」：preset_sources, window, invalidation, snapshot, tracking の一式。

## 5. Step 3: 評価軸設計

### 目的

3 軸（setup_maturity, timing, risk_reward）の
具体的な評価指標とスコアリング方法を決定する。

### 実施内容

#### 5.1 setup_maturity 指標のリサーチ

以下の観点で指標候補を洗い出す。

- このセットアップがエントリ可能状態に近づく際に、
  何が観測可能な形で変化するか
- その変化は日足 OHLCV から算出可能か
- 現行システムのフィールドで対応可能か、新規算出が必要か

典型的なカテゴリ:
- 出来高の変化（乾燥、急増、偏り）
- 価格と MA の位置関係
- ボラティリティの変化（収縮、拡大）
- 時間的要素（経過日数、持続期間）
- トレンド構造の維持度
- RS の変化

各候補について:
- 使用フィールド
- なぜこの指標がセットアップ成熟度を示すのかの根拠
- 重み配分の根拠
- スコアリングの方向性（線形、ベル型、閾値型）

#### 5.2 timing 指標のリサーチ

以下の観点で指標候補を洗い出す。

- このセットアップからの理想的なエントリ日に、
  何が当日の price action として観測されるか
- イベント型（cross-above, breakout）か状態型（zone, ratio）か
- 複数の timing 指標が同時に発火する日が最良のエントリ日か

典型的なカテゴリ:
- MA クロス / リクレームイベント
- 出来高の当日確認
- 引け位置の品質
- 短期構造のブレイク
- 需要の痕跡（PP, volume pattern）

#### 5.3 risk_reward 参照点の決定

以下を決定する。

- SL 参照点: セットアップが壊れたことを示す価格水準
  - 構造的参照（swing low, base 下限, pivot 下方等）
  - 既存フィールドで算出可能か
  - 新規インジケーターが必要か
  - pool_tracking で追跡すべきか

- TP 参照点: リワードの目標価格水準
  - 一次目標（保守的、到達確率が高い）
  - 二次目標（積極的、文脈情報）
  - フォールバック（一次目標を既に超えている場合）

- ATR バッファ: ノイズを避けるためのバッファ幅とその根拠

- エッジケースの処理方針

#### 5.4 データ充足性の検証

Step 5.1–5.3 で列挙した全指標について、
現行システムで算出可能かを検証する。

参照ドキュメント: `data_inputs_and_derivable_metrics_inventory.md`

分類:
- A: 既存フィールドでそのまま使える
- B: 既存 raw データから追加算出可能（indicator 層の追加）
- C: pool_tracking で追跡することで算出可能
- D: 現行データでは算出不可能

D に分類されたものは代替指標を検討するか、設計から除外する。
B に分類されたものは、汎用性があるか（他 signal でも使えるか）を判断し、
汎用性が高ければ indicator 層に追加、低ければ evaluator 内で算出する。

### 成果物

「評価軸設計書」：3 軸の指標一覧、重み、スコアリング方法、データ充足性評価。

## 6. Step 4: 統合とスコアリング設計

### 目的

3 軸スコアから entry_strength を算出する方法と、
表示閾値を決定する。

### 実施内容

#### 6.1 entry_strength 統合式

以下を決定する。

- 3 軸の重み配分
  - どの軸が最も時間感度が高いか
  - どの軸が「足切り」的な性質を持つか
  - セットアップの特性に応じた配分根拠

- floor gate の要否
  - 1 軸が極端に低い場合の処理
  - min_axis_threshold と capped_strength の値

原則:
- 重み配分はセットアップごとに異なりうる
  （pullback 系は timing 重視、breakout 系は maturity 重視、等）
- floor gate はシステム共通の仕組みとして提供し、
  閾値はシグナルごとに設定可能にする

#### 6.2 表示閾値

以下を決定する。

- Signal Detected 閾値（エントリ検討対象として強調表示）
- Approaching 閾値（監視対象として通常表示）
- Tracking（閾値以下、折りたたみ表示）

原則:
- 初期値は 50 / 35 / 0 を基本とする
- 検証データに基づいて調整する
- 閾値は config 化しておく

### 成果物

「統合設計書」：統合式、floor gate、表示閾値。

## 7. Step 5: システム整合性検証

### 目的

設計した Entry Signal が、既存システムの制約と矛盾しないことを確認する。

### チェックリスト

#### 7.1 データ整合性

- [ ] 使用する全フィールドが enriched snapshot に存在する
- [ ] 新規追加フィールドの算出式が明確である
- [ ] snapshot_at_detection の全フィールドが取得可能である
- [ ] pool_tracking フィールドの更新ロジックが明確である

#### 7.2 DB 整合性

- [ ] signal_pool_entry の UNIQUE 制約に問題がないか
- [ ] signal_evaluation の記録粒度が適切か
- [ ] 既存テーブルとの結合が必要な場合、結合キーが存在するか
- [ ] マイグレーション手順が明確か

#### 7.3 Runner 整合性

- [ ] 新 signal が既存の boolean signal と並走できるか
- [ ] pool 更新と evaluation の実行順序が正しいか
- [ ] パイプライン実行時と保存済み run 読み込み時の挙動が定義されているか

#### 7.4 preset 依存性

- [ ] preset_sources の各 preset が built-in か custom か
- [ ] custom preset のリネーム・削除に対する orphaned 処理が設計されているか
- [ ] preset_sources に指定した preset が持つ scan/annotation 条件を把握しているか

#### 7.5 UI 整合性

- [ ] 表示に必要な全フィールドが evaluation 結果に含まれるか
- [ ] signal ごとのセクション分けが既存 UI 構造と整合するか
- [ ] 同一銘柄が複数 signal に出現する場合の表示方針が決まっているか

#### 7.6 懸念事項の網羅

以下の観点で、設計上の懸念がないか最終確認する。

- pool の source of truth は明確か
- 再検出ルール（更新 vs 新規行）は明確か
- detection window の起算点は明確か
- signal 定義変更後の履歴整合性は確保されているか
- パフォーマンス上のボトルネックはないか

### 成果物

「整合性検証チェックリスト」：全項目の確認結果と対処方針。

---

## 8. Step 6: 実装計画書作成

### 目的

設計を実装可能なタスク単位に分解し、
依存関係と受け入れ基準を明確にする。

### 標準構成

```
1. 概要
   - 目的
   - 設計原則
   - 対象セットアップの要約

2. 前提条件と追加実装
   - indicator 層の追加
   - DB スキーマ
   - snapshot フィールド

3. Entry Signal 定義 (YAML)
   - 全設定の一括定義

4. 実装フェーズ
   Phase 1: 基盤層（DB + インジケーター）
   Phase 2: 評価エンジン（スコアリング + evaluator）
   Phase 3: Runner 統合
   Phase 4: UI 表示
   Phase 5: 検証

5. 依存関係図

6. リスクと対策
```

### タスク記述の標準フォーマット

各タスクは以下を含む。

- 対象ファイル / モジュール
- 変更内容の概要
- 必要な入力
- 受け入れ基準（testable な条件）
- 依存タスク

### 成果物

「実装計画書」：上記構成に従った文書。

---

## 9. Step 7: 実装・検証

### 実装

Phase 順に実装する。各 Phase 完了時に受け入れ基準を確認する。

### 検証

#### 遡及検証

過去データに対して新 signal を実行し、以下を分析する。

- entry_strength と forward return の相関
- 表示閾値の妥当性
- 各サブ指標の弁別力
- pool のライフサイクル（平均 active 日数、invalidation 率）

#### 運用検証

2 週間のライブ運用で以下を確認する。

- signal_evaluation テーブルのデータ品質
- UI の実用性
- pool 更新の安定性

### 成果物

「検証レポート」：分析結果と、breakpoints / weights / 閾値の調整推奨。

## 11. 次の Entry Signal 候補

Orderly Pullback Entry の検証完了後、以下の順序で展開を検討する。

1. **Base Breakout Entry**
   - tight base からの breakout を対象
   - VCS, contraction 期間, volume surge が主要指標
   - 検出とエントリが同日に近いため、detection_window は短い

2. **Structure Pivot Entry**
   - LL-HL 構造転換後の最初の breakout
   - structure_pivot 系フィールドが既に存在
   - recovery play のため、R/R の参照点が異なる

3. **Momentum Continuation Entry**
   - 強い上昇後の短期調整からの再加速
   - RS Acceleration, 4% bullish との親和性が高い
   - detection_window は非常に短い（2-3 日）

各候補は本文書の Step 1 から順に設計プロセスを回す。
