# プリセット CSV 出力機能 要件定義

## 1. 文書の目的

本書は、`Today's Watchlist` のプリセット CSV 出力機能を、現行の「手動ダウンロード中心」から、セットアップ該当銘柄の有無を短時間で把握しやすい形へ再設計するための要件を定義する。

最大の目標は次のとおり。

```text
「セットアップに該当する銘柄(エントリー候補)が存在するかを短時間で把握したい」
```

## 2. 背景

現行実装では、選択中の単一プリセットを UI のサイドバーから手動で CSV ダウンロードする方式になっている。

この方式には次の課題がある。

- 毎回明示的な UI 操作が必要
- 複数プリセットをまとめて確認しづらい
- 「候補があるかどうか」の即時確認より、「詳細をエクスポートする」用途に寄っている
- 起動後またはデータ更新後の自動確認フローに組み込みにくい

## 3. 機能の目的

本機能の目的は、ユーザーがその日の候補状況を短時間で把握できるようにすることである。

具体的には以下を満たす。

- 指定した複数プリセットについて、候補の有無を一覧で確認できる
- 必要であれば、各プリセットの候補銘柄も確認できる
- プリセットや出力対象は、ユーザーが容易に変更できる
- UI 依存を下げ、自動出力ワークフローを許容する

## 4. スコープ

本要件の対象は、`Today's Watchlist` に関連するプリセット CSV 出力機能の再設計である。

本要件に含むもの:

- 出力対象プリセットの定義方法
- CSV 出力タイミング
- 出力先フォルダ構成
- CSV フォーマット
- 同日再実行時の扱い

本要件に含まないもの:

- スキャンルール本体の変更
- Watchlist 判定ロジックの意味変更
- Entry/Exit 管理機能の追加

## 5. 基本方針

### 5.1 出力の主目的

CSV は「詳細一覧の保存」よりも先に、「候補が存在するかの確認」を主目的とする。

### 5.2 プリセット定義

出力対象プリセットおよび各プリセット内容は、ユーザーが容易に変更できる構成とする。

第一候補は設定ファイル管理とし、Git 管理可能な形を優先する。

### 5.3 実行方法

UI からの明示操作は必須としない。

データ取得完了後の自動出力を許容し、必要であれば UI 操作による補助実行も可とする。

### 5.4 保存方針

出力先フォルダは日付単位の `YYYYMMDD` とする。

起動ごとの `YYYYMMDD_HHMMSS` フォルダは採用しない。

同日内の再実行では既存ファイルを上書きし、フォルダやファイルの乱立を避ける。

## 6. 機能要件

### 6.1 出力対象プリセットの管理

システムは、CSV 出力対象とするプリセット群を設定ファイルで定義できなければならない。

最低限、以下を定義できること。

- プリセット名
- 対象 scan 名一覧
- annotation filter 一覧
- duplicate threshold
- duplicate subfilter 一覧
- CSV 自動出力対象に含めるかどうか

### 6.2 出力タイミング

システムは、少なくとも次のタイミングで CSV を出力できなければならない。

- 全データ取得と watchlist 生成が完了した後

補助要件:

- 将来的に UI 操作で手動再出力できてもよい
- ただし主ワークフローは自動出力を優先する

### 6.3 出力単位

システムは、複数プリセットの結果をまとめた CSV を出力しなければならない。

最小要件として、1 日あたり 1 つのサマリー CSV を出力する。

必要に応じて詳細 CSV を追加出力してもよい。

### 6.4 保存先

出力先は日付単位フォルダとする。

例:

```text
<export_root>/20260409/
```

同日内の再実行では同一ファイル名へ上書き保存する。

例:

```text
<export_root>/20260409/preset_summary.csv
<export_root>/20260409/preset_details.csv
```

### 6.5 上書きルール

システムは、同一日付フォルダ内の既存出力ファイルを上書きしなければならない。

履歴保持のための時刻付き別名ファイルは、本要件では採用しない。

## 7. CSV 要件

### 7.1 必須出力: `preset_summary.csv`

`preset_summary.csv` は、各プリセットの候補有無を最短で判断するためのサマリーとする。

最低限含める列:

- `trade_date`
- `output_date`
- `preset_name`
- `has_candidates`
- `candidate_count`
- `top_tickers`
- `selected_scan_names`
- `selected_annotation_filters`
- `duplicate_threshold`

列の意味:

- `has_candidates`: 候補が 1 件以上あるかの真偽
- `candidate_count`: そのプリセット条件で候補となった銘柄数
- `top_tickers`: 上位候補ティッカーを短い文字列で連結したもの

### 7.2 任意出力: `preset_details.csv`

必要に応じて、現行の詳細 CSV 相当の内容を別ファイルとして出力してよい。

想定内容:

- `Duplicate Tickers`
- 各 scan card ごとの hit tickers

ただし、詳細 CSV は補助出力であり、主目的は `preset_summary.csv` による即時把握とする。

## 8. 候補判定の考え方

各プリセットの候補判定は、当該プリセットに定義された条件を適用した watchlist 結果に基づく。

初期方針:

- selected scan names を適用する
- selected annotation filters を適用する
- duplicate threshold を適用する
- その結果として残った候補数を `candidate_count` とする
- `candidate_count > 0` のとき `has_candidates = true` とする

`top_tickers` は、候補集合のうち並び順上位の一定件数を連結して格納する。

件数上限は実装時に設定化してもよい。

## 9. ユーザー変更容易性

本機能では、ユーザーがプリセットや出力対象を変更しやすいことを重視する。

そのため次を満たすこと。

- 出力対象プリセット群は設定ファイルで変更可能
- 各プリセットの scan 構成も設定ファイルで変更可能
- コード修正なしで、対象プリセットの追加・削除・並び替えが可能

## 10. UI 要件

本機能の成立に UI 操作は必須ではない。

ただし、実装コストや保守性の観点から必要であれば、補助的に以下を追加してもよい。

- 手動再出力ボタン
- 出力先表示
- 最終出力時刻表示

ただし、主ワークフローは「起動後またはデータ取得完了後の自動出力」とする。

## 11. 非機能要件

- 出力処理は watchlist 生成後の既存結果を再利用し、過剰な再計算を避けること
- 出力ファイル名と保存場所は人間が見て即理解できること
- 同一日の確認場所は常に 1 フォルダに集約されること
- プリセット数が増えても保守しやすい構成であること

## 12. 受け入れ条件

以下を満たした場合、本要件を満たすものとする。

1. 設定ファイルで指定した複数プリセットについて、1 回の実行で CSV が出力される
2. 出力先が `YYYYMMDD` フォルダである
3. 同日再実行時に既存ファイルが上書きされる
4. `preset_summary.csv` を見れば、各プリセットについて候補有無と件数を短時間で判断できる
5. プリセット定義や出力対象変更にコード修正を必須としない

## 13. 実装検討メモ

現状の実装は「選択中単一プリセットを UI からダウンロード」する設計である。

本要件では、これを以下へ寄せる想定とする。

- 設定ファイルベースの出力対象プリセット管理
- watchlist 生成完了後の自動バッチ出力
- サマリー CSV を主、詳細 CSV を従とする二層構成

以上により、ユーザーは毎回 UI で個別出力しなくても、その日の候補状況を即確認できる。
